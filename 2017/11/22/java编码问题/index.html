cmd控制台添加utf-8编码方式

按下快捷键Win+R，输入regedit打开注册变编辑器，依次找到
HKEY_LOCAL_MACHINE\Software\Microsoft\Command Processor\
右键新建一个字符串值(S),取名叫Autorun,值设为@chcp 65001>nul
确定保存退出
重新运行程序即可

控制台更换编码方式命令

chcp <字符集编码>

UTF8 与 UTF8 +BOM 区别

BOM是Byte Order Mark（定义字节顺序），在网络传输中分两种传输顺序：Big-Endian和Little-Endian（俗称大头和小头）

在UCS (即Unicode，Unicode的学名是”Universal Multiple-Octet Coded Character Set”，简称为UCS) 编码中有一个叫做”ZERO WIDTH NO-BREAK SPACE”的字符，它的编码是FFFE。（FFFE在UCS中是不存在的字符，故可用于表示标签而不会导致歧义）
UCS规范建议我们在传输字节流前，先传输字符”ZERO WIDTH NO-BREAK SPACE”。这样如果接收者收到FEFF，就表明这个字节流是Big-Endian的；如果收到FFFE，就表明这个字节流是Little- Endian的。因此字符”ZERO WIDTH NO-BREAK SPACE”又被称作BOM。

big endian和little

endian是CPU处理多字节数的不同方式。例如“汉”字的Unicode编码是6C49。写到文件里时，如果将6C写在前面，就是big endian。如果将49写在前面，就是little endian。

而utf-8即Unicode的一种目前使用最广的实现方式，每次传输8位的数据，故也就不存在多字节传输顺序的问题，即编码方式可以不带BOM。
但也可以用BOM来表示编码方式，windows就是这么干的。
字符”ZERO WIDTH NO-BREAK SPACE”的UTF-8编码是EF BB BF。所以接收的字节流开头是EF BB BF，即表明是utf-8编码。

ANSI、Unicode、GBK、GB2312 和 UTF-8 字符编码的区别联系

ANSI即为ASCII码；
GB2312是ASCII的中文扩充；分高低字节。
GBK是GB2312的扩充，第一个字节大于127，即超出ASCII的范围即开始表示扩充的符号。
Unicode即万国码，一般情况下硬性要求两个字节表示一个符号；且规定两个字节就是“一个字符”；unicode中一个中文字符占2个字节。
UTF-8即Unicode的一种目前使用最广的实现方式，每次传输8位的数据，UTF-16就是每次16个位；它可以使用1~4个字节表示一个符号，英文使用一个字节编码，中文使用三个字节来编码。

java编码问题

编译转码

因为，Java是用Unicode表示的语言，所以Java编译之前，要把源文件转成unicode编码，再进行编译。
此时会存在两个编码方式之间的统一问题：
首先是源文件的编码方式，操作系统默认的编码方式
一般情况下，中文操作系统默认编码方式是GBK，英文情况下是Utf-8
编译器会按照操作系统的编译方式将源文件转换成Unicode，如果两者不统一，最后编译生成的unicode就是乱码的

输出转码

代码以Unicode的形式保存再内存中，理论上讲，在向标准输出（控制台）输出时，jvm又进行了一次转码，采用操作系统默认的编码方式，最后输出到控制台上。
此时也应该统一两个编码方式：
控制台显示的编码，操作系统默认的编码

但是我实际操作过程中，无论控制台的编码方式是什么，只要确保上一步编译转码是正确的，就可以正确无乱码的输出，含中文情况。
也就是说我的输出时是按照控制台的编码方式将Unicode转换的

总之
控制好各处的编码方式，出现乱码及时分析原因进行修正。
